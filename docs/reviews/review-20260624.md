# AI Trader — Architecture Review (2026-05-24)

A strict lead-engineer review of the codebase as a **foundation** for the stated goal: a
semi-autonomous AI assistant for Polymarket that (phase 1) finds, analyzes and risk-assesses
bets, and (phase 2) eventually places orders, manages a portfolio, and reports performance.

Scope reviewed: the LangGraph graph, protocols, node components, domain/state models, the
skills layer, the composition root, clients, config, and the eval harness. The docs
(`DESIGN.md` / `PLAN.md`) are accurate — the code genuinely matches them, which is rare.

---

## Verdict

- **As a research assistant (phase 1)** — solid, professionally assembled. **7.5/10.**
  Clean DI, protocol seams, no import-time side effects, skills-as-specialization, an eval
  harness behind a vendor-neutral backend protocol.
- **As a foundation for semi-autonomous trading (phase 2)** — **5/10.** The architecture is
  built around "one conversational turn → structured answer" and lacks the seams that
  distinguish a trading agent from a chatbot: domain state (positions / balance / P&L), a
  transactional audit trail, and the ability to pause for human-in-the-loop approval before
  an irreversible action. This is not a "finish the TODOs" gap — it changes the load-bearing
  axis of the system.

---

## Critical gaps (for the trading goal)

### 1. The verifier checks the label, not the contents — and you trade on the contents
`core/components/verifier/__init__.py` guarantees only that every referenced `market_id`
appeared in tool output. The **financially material numbers** — `implied_probability`,
`fair_probability`, `edge` — are **never verified**. The entire value proposition of
`analyze` is "implied vs. fair." The responder can rewrite implied from 0.62 to 0.72 or
fabricate a fair value, and the verifier passes it because the id is real. An
anti-hallucination check that validates the id but not the prices is a false sense of trust.
The eval `grounding` evaluator (`tests/eval/evaluators.py`) mirrors the same narrow check, so
the metrics inherit the same false confidence.

→ The verifier should reconcile `implied_probability` against what `polymarket_*` actually
returned for that `market_id` (this is as deterministic as the id check). Fair value is
model-derived by nature, but it must be required to be grounded in tool output, not invented.

### 2. The numbers you trade on are produced by the WEAK model
The key structural risk. The `planner` (strong model) reasons in free text. The structured
fields — including `fair_probability`, `implied_probability`, `edge` — are filled by the
`responder` running on the **weak** model (`core/bootstrap.py`). So the strong model says one
thing in prose, and the weak model transcribes it into the number you act on. Two LLM hops
with information loss on exactly the fields where the cost of error is highest, plus drift
between the `summary` and the structured fields.

→ On the "answer" branch, the structured output should come **directly from the strong
model** (via `with_structured_output` or a final-answer tool on strong), removing the
strong→weak handoff. A separate weak responder re-synthesizing financial numbers is an
anti-pattern.

### 3. There is no domain state — only the conversation
`AgentState` (`core/models/schemas.py`) is purely conversational (`messages`, `skill`,
`iteration`, verdicts, `result`). There is no notion of positions, balance, watchlist, prior
recommendations, or P&L. The checkpointer stores the **conversation keyed by `thread_id`** —
the right tool for chat memory and the **wrong** tool for account/portfolio state, which is
cross-thread, transactional, and queryable. The moment `/buy` or `/watch` exists, you need a
separate datastore (positions / orders / runs-audit) that the current design does not
anticipate. This is the biggest foundational gap: the design has no "conversation state ⟂
account state" boundary.

→ Introduce this seam now (a portfolio/order repository behind a protocol, as already done
for `Agent` and the clients), even if the first implementation is a stub. Otherwise phase 2
forces a core rewrite.

### 4. Human-in-the-loop before a buy is incompatible with the current transport
`PLAN.md` says the guard becomes the approval gate via `interrupt()`. But the `guard`
(`core/components/guard/__init__.py`) only returns ALLOW/BLOCK — there is no third "pause for
human approval" path — and `app/main.py` plus `ReActAgent.invoke()` are strictly one-shot
request/response with no resume mechanism. `interrupt()` for an **irreversible money action**
is not a one-liner: it breaks the HTTP layer, the Telegram layer, and the `invoke()`
signature (you need a two-phase / resumable call). This is the most safety-critical seam and
it is currently conceptually incompatible with how the transport works. Decide on this
**before** building `/buy`, because it dictates the API design.

---

## Substantial, but not blocking

### 5. The guard is LLM theater on the hot path
The `guard` runs an LLM judge on **every** batch of tool calls, while — by `PLAN.md`'s own
admission — being "permissive for read-only tools," i.e. rubber-stamping "allow" at the cost
of latency and money. For read-only this should be a deterministic policy (a tool allowlist);
the LLM judge should only engage once mutating tools exist. Today a simple `/find` pays:
`selector` (LLM) + N×`planner` (LLM) + N×`guard` (LLM) + `responder` (LLM) — 5–7 calls, of
which the guard (and often the selector) is pure tax. The "one LLM node per concern" pattern
scales poorly into latency.

### 6. `temperature=0.4` everywhere, including numeric extraction
`bootstrap.py` uses a single temperature for all models. For the responder (extracting
`fair_probability` / `implied_probability`) and for eval reproducibility, non-zero temperature
is a liability: the numbers are not reproducible. At minimum use `temperature=0` for
structured output.

### 7. Brittle external IO
`PolymarketClient` (`core/clients/polymarket/__init__.py`) and Tavily are single-shot, with no
retries/backoff, and errors are returned **as a string** into the model
(`"Polymarket search failed: ..."`). The model may "reason around" that string and
hallucinate. Tolerable for an assistant, not for autonomy. Add retries and signal failures
into the graph explicitly rather than into the model's prompt.

### 8. `recursion_limit = max*6+10` is topology-coupled magic
`react.py` encodes "nodes per iteration" as a number. If the graph topology changes, this
silently breaks. Minor, but it couples the safety net to the topology.

## What is genuinely good (keep)

- **Composition root and protocols** (`core/models/protocols.py`, `core/bootstrap.py`) —
  tools are injected, nodes sit behind Protocols, everything is swappable. The right skeleton.
- **Skill-as-specialization** — adding a skill is one module, no graph changes. Storing the
  skill **name** in state (not the object, because the checkpointer serializes state) is the
  correct, documented call.
- **`referenced_market_ids()` as a polymorphic method** — the verifier stays schema-agnostic.
  The idea is right; it just needs to be extended from ids to prices (see #1).
- **Eval behind an `EvalBackend` protocol** — vendor-neutral, with case distillation from
  traces. More mature than most projects of this size.

---

## Priorities (if I owned this)

1. **Before any phase-2 work:** move the "answer" branch onto the strong model (kill the
   strong→weak handoff on numbers) and extend the verifier to reconcile `implied_probability`
   (#1, #2). This raises trust in what already exists.
2. **Design the domain layer now:** `PortfolioRepository` / `OrderRepository` behind a
   protocol + a `runs` audit table. Stubs are fine — the point is to make the boundary exist
   in the architecture (#3).
3. **Rework the transport for resumable execution** (two-phase invoke / streaming); otherwise
   human-in-the-loop `interrupt()` cannot be embedded (#4).
4. Make the guard deterministic for read-only; engage the LLM judge only for mutating tools
   (#5).
5. `temperature=0` on structured output (#6).
