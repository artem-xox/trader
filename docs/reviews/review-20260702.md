# AI Trader — Architecture Review (2026-07-02)

A follow-up strict lead-engineer review of the codebase as a **foundation** for the stated
goal: a semi-autonomous AI assistant for Polymarket that (phase 1) finds, analyzes and
risk-assesses bets, and (phase 2) eventually places orders through the agent.

Scope reviewed: the LangGraph graph, protocols, node components, domain/state models, the
skills layer, the composition root, clients (Gamma, CLOB, Tavily), the FastAPI app, the
Telegram bot, the eval harness, the docs, and the previous review
([review-20260624.md](review-20260624.md)).

---

## Verdict

- **As a research assistant (current phase)** — **7/10.** The skeleton remains one of the
  cleanest of its size: composition root, protocol seams, skills-as-specialization,
  vendor-neutral eval. Streaming and the eval harness have matured noticeably. But this
  review found a bug that silently degrades every multi-turn conversation — it drags the
  score below what the architecture otherwise deserves.
- **As a foundation for semi-autonomous trading (phase 2)** — **5.5/10.** The three
  load-bearing seams from the previous review are still missing (domain state, resumable
  transport, number-level verification), and the CLOB metrics carry a units bug that
  undermines exactly the cost model the whole Phase 1–3 roadmap is built on.

## New findings — bugs

### 1. `iteration` never resets between turns — the budget leaks and the verifier switches off (critical)

`core/components/planner/__init__.py` does `state.get("iteration", 0) + step`, and
**nothing ever resets the counter**. The checkpointer persists *all* state channels per
`thread_id`, not just `messages`. So in a live Telegram chat the
`AGENT_MAX_ITERATIONS = 8` budget is a budget **per thread, not per turn**:

- Turn 1 (`/find ...`) consumes, say, 6 iterations. Turn 2 starts at `iteration=6` and
  gets 2 planner steps. Turn 3+ gets exactly one batch of tool calls before being forced
  into the responder.
- Worse: `_route_after_verifier` (`react.py`) converts `REVISE` into `OK` once the budget
  is exhausted — meaning **the anti-hallucination guarantee, the system's core trust
  mechanism, silently switches off** in every long-lived chat.

This is invisible degradation: no error, no log line — answers just get shallower and
invented ids stop being caught. Neither the tests nor the eval harness can see it because
everything is single-turn (the eval runner uses a fresh `uuid` thread per case). The fix
is trivial — reset `iteration` (and the verdicts) in the selector node, which already runs
first on every turn — plus one multi-turn regression test.

### 2. CLOB: shares vs USD confusion — the cost-model foundation computes in the wrong units

In the Polymarket CLOB book, `size` is denominated in **shares (outcome tokens), not
dollars**. The USD value of a level is `price × size`. But in `core/clients/clob/__init__.py`:

- `BookLevel.size` is commented as "USD notional" — wrong.
- `depth_within_2_ticks` sums raw `size` values and the tool reports the result as
  `depth_within_2_ticks_usd`. At price 0.5 that overstates depth 2×, at 0.05 — 20×. The
  model reasons about liquidity from a number whose error grows the further the market is
  from 50/50 — i.e. exactly on the longshots where liquidity matters most.
- `slippage_from_book` mixes units inside one function: `min(lv.size, size_usd - filled)`
  compares shares against dollars, while `cost += take * lv.price` treats `take` as
  shares. The tests (`tests/unit/test_clob_client.py`) encode this confusion rather than
  catch it. The function is currently dead code (only tests call it), but it is precisely
  the function Phase 1 plans to build EV and sizing on. If it isn't fixed before Phase 1,
  a units bug becomes financially material.

### 3. `realised_vol_1h` is almost always `None` — and it isn't volatility

`orderbook_snapshot` requests `prices_history(interval="1h", fidelity=60)` — fidelity is
the resolution in minutes, so a one-hour window yields 1–2 points, and `realised_vol`
returns `None` for `len < 2`. The field is most likely permanently empty in production —
worth confirming with a live test. Conceptually: the stdev of **price levels** is not
realised volatility (that is computed over returns/increments), and as implemented the 1h
and 24h numbers are not comparable across window sizes.

## Architecture assessment against the goal

### What still blocks "phase 2" (buying through the agent)

Repeating the previous review because none of it has resolved, and the roadmap is heading
straight at it:

1. **Domain state.** There is still no "conversation state ⟂ account state" seam.
   Watchlists, a recommendation history, positions — nowhere to put them. And this blocks
   more than trading: **Phase 4 (calibration, Brier) is impossible without a log of the
   agent's own recommendations.** Today the only record of what the agent advised is
   LangSmith traces. A cheap append-only `recommendations` table (market_id, stance,
   fair_prob, implied_prob, timestamp) is the highest-ROI step of all the "independent of
   phases" items: it turns Phase 4 from "eval on historical data" into "calibration of the
   agent's actual calls".
2. **Resumable transport.** SSE is the right step, but `interrupt()` for order
   confirmation requires: a checkpointer that survives restarts (still `InMemorySaver`), a
   `POST /agent/resume` endpoint, and callback buttons in Telegram. None of this exists,
   and PLAN honestly says so. Good that it's recorded; bad that the API design still isn't
   fixed — it will shape `/agent/stream`.
3. **Number verification.** `MarketAnalysis.implied_probability` can be reconciled
   deterministically against what `polymarket_market` returned — same complexity as the id
   check, still not done. Today the responder can rewrite implied from 0.62 to 0.72 and the
   verifier passes it. For a system whose value proposition is "implied vs fair", this is
   trust-model hole number one.

### Phase 0/1 half-landed — the data flows in, the contract doesn't capture it

The CLOB client and `polymarket_orderbook` are built (good), but: the `analyze` planner is
required to fetch the order book, while `MarketAnalysis` has **no execution fields at
all** — spread/depth influence only the prose. Which means: (a) the verifier can't check
it, (b) the eval can't measure it, (c) the formatter can't render it. The data flows into
the context and dissolves. This is expected per the roadmap (the fields arrive in
Phase 1), but right now the project is in an "investment made, payoff not wired up" state —
Phase 2 should not start until Phase 1 closes the loop data → schema → verifier → eval.

### Guard: dead code and docs that lie

The guard is an allow-all stub (`core/components/guard/__init__.py`). Dropping the LLM tax
was the right call, but:

- `BASE_GUARD_PROMPT`, `GuardJudgment`, and the `Skill.guard_prompt` field (populated by
  both skills) are dead code that looks alive. A new developer (or an LLM agent) will
  assume the guard prompts do something.
- The middle ground between a stub and an LLM judge — a **deterministic allowlist** (the
  skill's `tools` already define what's permitted; the guard could check that the proposed
  calls ⊆ that set) — is ~10 lines and would provide a real guarantee instead of a
  decorative one.

### Documentation consistency — a regression

The previous review explicitly praised "the docs match the code, which is rare". One week
later, drift has crept in:

- [DESIGN.md](../DESIGN.md) §2: guard listed as "weak model" (actually no LLM), responder
  as "weak" (actually strong); §5 lists 3 of the 8 tools (missing orderbook, calculator,
  current_datetime, web_fetch, think); §13 references `core/prompts.py`, which doesn't exist.
- [PLAN.md](../PLAN.md): "the guard runs an LLM judge" — no longer true; Phase 0 is written
  as future work although commit `743249b` shipped it (client, tool, metrics, tests — all
  in main).

Individually minor, but the sum means DESIGN.md is no longer a reliable map — and for a
project that clearly invests in docs as the interface for future sessions/agents, that is
a regression worth fixing with the same discipline as code.

### Other substantial items

- **Race on `thread_id`.** Two quick messages in one chat → two concurrent `ainvoke` runs
  on the same `thread_id` → interleaved history in the checkpointer. aiogram handles
  updates concurrently and serializes nothing. The bot needs a per-chat lock (or queue) —
  the PLAN open question about "rate limiting" is really about this.
- **Unbounded history growth.** No trimming, no summarization: a long chat means a growing
  planner prompt every turn (and growing cost). The LangGraph answer is `trim_messages`
  before the planner or a summarization node; the roadmap lacks this item and it will bite
  before `BaseStore` does.
- **The verifier grounds against the whole thread history.** `_market_ids_seen` scans all
  messages — an id fetched three turns ago (possibly a now-closed market with a stale
  price) is still valid today. Arguably acceptable for ids, but once price verification
  lands, the window must narrow to the current turn.
- **Minor:** `fair_probability`/`implied_probability` lack `ge=0, le=1` Pydantic bounds
  (free protection); API-key comparison isn't `hmac.compare_digest`; unknown slash commands
  in the bot are silently ignored; the eval runner reaches into private
  `agent._graph`/`._settings` (deliberate, but it's why the recursion_limit formula
  multiplied).

## What is genuinely good (and got better)

- **Streaming is exemplary layering:** semantic `ProgressEvent`s in core, SSE in app,
  rendering in the client — not a single emoji in the core. It is the best new thing in
  the codebase, and half the groundwork for future HITL.
- **Responder on the strong model** — the previous review's top numeric risk is closed,
  plus a tidy `structured_call` with a recovery path instead of LCEL wrappers (with a
  well-argued docstring).
- **The eval harness is still the strongest part of the project:** 6 evaluators including
  a trajectory-based `tool_use`, judge at t=0 with a reasoning-first schema, YAML cases
  under review, case distillation from traces. More mature than most production agents.
- **The "a think step doesn't spend budget, but a think chain does" discipline** — small,
  well-reasoned, well-commented.
- **`safe_eval`** — a correct AST whitelist with `9**9**9` protection.

## Priorities (if I owned this)

1. **Reset `iteration` in the selector + a multi-turn regression test** — an hour of work,
   fixes invisible degradation and the silent verifier shutdown. Until this lands, any
   conclusion drawn from live bot usage is skewed.
2. **CLOB units (shares vs USD) + fix `realised_vol`** — before Phase 1 starts, because
   EV/sizing will stand on these functions.
3. **Verify `implied_probability` in the verifier** — the same deterministic mechanics as
   the id check; closes point #1 of two consecutive reviews.
4. **A `recommendations` table (append-only, SQLite is fine)** — unlocks real calibration
   in Phase 4 and becomes the first brick of the domain layer from the previous review's
   point #3.
5. **Sync DESIGN/PLAN with the code + remove the dead guard code** (or replace the stub
   with a deterministic allowlist).
6. Then proceed with the roadmap as written: the plan itself is good (the
   analyze → find → distribute sequencing around a shared cost model is right); the problem
   is not the plan but the items above eroding its foundation.
